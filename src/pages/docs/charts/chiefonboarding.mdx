# ChiefOnboarding

Employee onboarding automation platform. ChiefOnboarding lets HR teams define onboarding sequences for new hires â€”
assigning tasks, collecting documents, sending introductions, and integrating with tools like Slack and Google
Workspace. All onboarding data and workflow definitions are stored in PostgreSQL.

## Key Features

- **Onboarding sequences** â€” ordered tasks, automated triggers, and conditional workflows
- **Integrations** â€” Slack, Google Workspace, email notifications, and webhook support
- **Multi-role access** â€” admin, manager, and buddy role assignments
- **PostgreSQL backend** â€” bundled subchart or external database
- **Django application** â€” secure SECRET_KEY management for sessions and CSRF protection
- **Ingress support** â€” TLS via cert-manager with configurable ingress class

## Installation

**HTTPS repository:**

```bash
helm repo add helmforge https://repo.helmforge.dev
helm repo update
helm install chiefonboarding helmforge/chiefonboarding
```

**OCI registry:**

```bash
helm install chiefonboarding oci://ghcr.io/helmforgedev/helm/chiefonboarding
```

### ChiefOnboarding Configuration

| Parameter                           | Type    | Default      | Description                                                              |
| ----------------------------------- | ------- | ------------ | ------------------------------------------------------------------------ |
| `chiefonboarding.port`              | integer | `8000`       | Internal HTTP port (Django development server).                          |
| `chiefonboarding.baseUrl`           | string  | `""`         | Public base URL of the instance (e.g. `https://onboarding.example.com`). |
| `chiefonboarding.secretKey`         | string  | `""`         | Django SECRET_KEY for session signing and CSRF protection.               |
| `chiefonboarding.existingSecret`    | string  | `""`         | Existing Kubernetes Secret containing the Django secret key.             |
| `chiefonboarding.existingSecretKey` | string  | `secret-key` | Key inside the existing secret holding the Django SECRET_KEY.            |
| `chiefonboarding.extraEnv`          | array   | `[]`         | Extra environment variables for integrations and advanced configuration. |

<Callout type="warning" title="Set baseUrl to your public URL">
  `chiefonboarding.baseUrl` must be set to your actual public URL before deploying. Django uses this value to generate
  links in email notifications and onboarding task messages. Without it, all links sent to new hires will be empty or
  incorrect.
</Callout>

<Callout type="danger" title="Always set secretKey explicitly">
  If `chiefonboarding.secretKey` is empty, a Django SECRET_KEY is auto-generated at startup. If the pod is recreated
  with a new key, **all user sessions are invalidated, all pending password reset links expire, and all CSRF tokens
  become invalid**. Always provide a stable, explicit key or use `chiefonboarding.existingSecret`.
</Callout>

<Callout type="tip" title="Integrations via extraEnv">
  ChiefOnboarding integrations (Slack, Google Workspace, email) are configured via environment variables. Use
  `chiefonboarding.extraEnv` with `secretKeyRef` to inject integration tokens securely. See the [ChiefOnboarding
  integration documentation](https://docs.chiefonboarding.com/docs/integrations/slack/) for the full list of required
  variables per integration.
</Callout>

## Common Issues

<Callout type="danger" title="Users cannot log in after upgrade">
  If users cannot log in after a Helm upgrade, the Django `SECRET_KEY` may have changed. Check that
  `chiefonboarding.secretKey` or `chiefonboarding.existingSecret` is set and matches the value used during the previous
  deployment.
</Callout>

<Callout type="tip" title="Create the first admin user">
  ChiefOnboarding does not ship with a default admin account. After the first deployment, access the instance and
  complete the initial setup wizard to create your admin credentials. The setup wizard is only shown when no admin
  account exists.
</Callout>
