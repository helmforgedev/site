# MySQL

<span data-i18n="docsCharts.mysql.intro">
  Production-ready MySQL deployment with support for standalone and source-replica replication architectures.
</span>

## <span data-i18n="docsCharts.heading.keyFeatures">Key Features</span>

- <span data-i18n="docsCharts.mysql.feature.standaloneReplication">
    <strong>Standalone and replication</strong> — Single instance or source-replica topology
  </span>
- <span data-i18n="docsCharts.mysql.feature.automaticInit">
    <strong>Automatic initialization</strong> — Init scripts and custom configuration
  </span>
- <span data-i18n="docsCharts.mysql.feature.backup">
    <strong>Backup support</strong> — Configurable backup CronJobs to S3-compatible storage
  </span>
- <span data-i18n="docsCharts.mysql.feature.metrics">
    <strong>Metrics</strong> — Prometheus exporter with ServiceMonitor
  </span>
- <span data-i18n="docsCharts.mysql.feature.security">
    <strong>Security</strong> — Non-root containers, network policies, TLS support
  </span>
- <span data-i18n="docsCharts.mysql.feature.storage">
    <strong>Persistent storage</strong> — Configurable PVCs with storage class selection
  </span>

```bash

helm install my-mysql helmforge/mysql
```

```yaml
  rootPassword: 'my-secret-password'
  database: myapp
  username: myuser

  persistence:

  enabled: true
  serviceMonitor:
    enabled: true
```

<div data-tab-panel="replication">

```yaml
# values.yaml
architecture: replication

auth:
  rootPassword: 'my-secret-password'
  database: myapp
  username: myuser
  password: 'user-password'
  replicationPassword: 'repl-password'

primary:
  persistence:
    size: 20Gi

secondary:
  replicaCount: 2
  persistence:
    size: 20Gi
```

</div>

```yaml
  enabled: true
  schedule: '0 3 * * *'

    endpoint:

    secretKey: wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY
  retention:
    days: 30
```

</div>

## <span data-i18n="docsCharts.heading.upgradeNotes">Upgrade Notes</span>

<Callout type="warning" title="Password persistence">
  <span data-i18n="docsCharts.mysql.upgrade.callout">
    MySQL stores the root password during initial setup. On upgrade, ensure `auth.rootPassword` matches the original
    value or the pod will fail to start with an authentication error.
  </span>
</Callout>

- <span data-i18n="docsCharts.mysql.upgrade.note1">
    When enabling replication on an existing standalone instance, a full re-initialization is required
  </span>
- <span data-i18n="docsCharts.mysql.upgrade.note2">Backup CronJob changes take effect on the next scheduled run</span>
- <span data-i18n="docsCharts.mysql.upgrade.note3">
    Secondary replicas are read-only; write attempts return an error
  </span>

## <span data-i18n="docsCharts.heading.commonIssues">Common Issues</span>

<Callout type="danger" title="Pod stuck in CrashLoopBackOff after upgrade">
  <span data-i18n="docsCharts.mysql.issue.crashloop">
    If the root password in values doesn't match the existing data directory, MySQL refuses to start.
    Check logs with `kubectl logs <pod>` and verify the password matches.
  </span>
</Callout>

<Callout type="tip" title="Replication lag monitoring">
  <span data-i18n="docsCharts.mysql.issue.replicationLag">
    Enable metrics and monitor `mysql_slave_status_seconds_behind_master`. Values consistently above 0 indicate the
    replica cannot keep up — consider increasing replica resources or reducing write load.
  </span>
</Callout>
